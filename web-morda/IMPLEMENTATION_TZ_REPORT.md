# Документация по реализации ТЗ «Книга рецептов»

Ниже зафиксировано, как текущее фронтенд-приложение реализует пункты ТЗ, какие места в коде за это отвечают, на что стоит обратить внимание и почему решение сделано именно так. Документ описывает **фактическое состояние текущей реализации**, а не идеальную целевую версию.

## Нефункциональные требования

### НФ-1. Веб-приложение с отдельной серверной частью
- **Ключевое место:** `src/api.config.ts`, `src/cookbookApi.ts`, `src/features/CookbookApp.tsx`
- **На что обратить внимание:** фронтенд работает как отдельный клиент и полностью зависит от backend API по `http://localhost:5284`.
- **Как реализовано:** приложение сделано как React SPA на Vite. Данные не хранятся локально как источник истины, а загружаются и изменяются через HTTP-запросы к серверу.
- **Почему так:** это напрямую соответствует требованию о клиенте с отдельной серверной частью и позволяет не дублировать бизнес-логику на фронтенде.

### НФ-2. Хранение данных на стороне сервера
- **Ключевое место:** `src/cookbookApi.ts`
- **На что обратить внимание:** фронтенд не управляет хранилищем и не знает, БД это или файлы.
- **Как реализовано:** все операции чтения и изменения продуктов и блюд выполняются через backend API.
- **Почему так:** в рамках фронтенда корректно опираться на сервер как на единственный источник данных; способ хранения скрыт за API.

### НФ-3. Серверная часть должна предоставлять API
- **Ключевое место:** `swagger.json`, `src/api.types.ts`, `src/cookbookApi.ts`
- **На что обратить внимание:** фронтенд построен по существующему OpenAPI-контракту.
- **Как реализовано:** типы запросов/ответов и набор методов клиента сформированы по `swagger.json`; используются endpoint'ы `/api/products`, `/api/products/{id}`, `/api/dishes`, `/api/dishes/{id}`.
- **Почему так:** это снижает риск расхождения между UI и backend и делает интеграцию предсказуемой.

### НФ-4. Код должен позволять писать тесты
- **Ключевое место:** `src/cookbookApi.ts`, `src/api.http.ts`, `src/features/CookbookApp.tsx`
- **На что обратить внимание:** тестовая инфраструктура пока не добавлена, но код разнесен по слоям.
- **Как реализовано:** API-слой, типы и UI разделены; HTTP-ошибки вынесены в `ApiError`, а запросы централизованы в `requestJson`.
- **Почему так:** такое разделение упрощает написание unit-тестов для утилит и API-обвязки, а также интеграционных/UI-тестов поверх экрана.

## 1. Управление продуктами

### 1.1 Создание продукта
- **Ключевое место:** `src/features/CookbookApp.tsx`, блок формы `Create Product`; `src/cookbookApi.ts#createProduct`
- **На что обратить внимание:** поля названы и сериализуются по контракту backend; фотографии сейчас вводятся как список ссылок/строк, а не upload файлов.
- **Как реализовано:** форма позволяет задать название, фотографии, калорийность, БЖУ, состав, категорию, cooking type и dietary flags. После валидации данные отправляются в `POST /api/products`.
- **Почему так:** UI строился строго по текущему `swagger.json`, где фотографии представлены строками, а не multipart-файлами.

### 1.2 Отображение продуктов
- **Ключевое место:** `src/features/CookbookApp.tsx`, режим `list` для `products`
- **На что обратить внимание:** фильтрация и сортировка делаются через query-параметры API, не локально.
- **Как реализовано:** добавлены фильтры по категории, cooking type, флагам, поиск по названию и сортировка по `SortBy`/`SortDirection`. Список рендерится таблицей.
- **Почему так:** backend уже предоставляет нужные query-параметры, поэтому фронтенд не дублирует правила фильтрации и сортировки.

### 1.3 Просмотр продукта
- **Ключевое место:** `src/features/CookbookApp.tsx`, режим `details` для `products`
- **На что обратить внимание:** детальная загрузка идет отдельным запросом по `id`.
- **Как реализовано:** при открытии карточки продукта вызывается `GET /api/products/{id}` и отображаются ключевые атрибуты, включая даты.
- **Почему так:** отдельная загрузка деталей не перегружает список и позволяет работать даже если список содержит усеченные данные.

### 1.4 Редактирование продукта
- **Ключевое место:** `src/features/CookbookApp.tsx`, режим `edit` для `products`; `src/cookbookApi.ts#updateProduct`
- **На что обратить внимание:** форма редактирования заполняется после загрузки продукта по `id`.
- **Как реализовано:** при переходе в edit-режим фронтенд получает продукт, переносит значения в форму и сохраняет изменения через `PUT /api/products/{id}`.
- **Почему так:** это гарантирует, что редактируется актуальная серверная версия объекта, а не устаревшая копия из списка.

### 1.5 Удаление продукта
- **Ключевое место:** `src/features/CookbookApp.tsx#removeProduct`, `src/api.http.ts`
- **На что обратить внимание:** ограничение на удаление продукта, который используется в блюде, сейчас зависит от backend-ответа `409`.
- **Как реализовано:** фронтенд вызывает `DELETE /api/products/{id}`; если сервер возвращает ошибку, она показывается через `ProblemDetails`.
- **Почему так:** проверка зависимости продукта от блюд должна быть авторитетно выполнена на сервере.
- **Статус относительно ТЗ:** **частично реализовано**. Сообщение о невозможности удаления выводится, но отдельного списка блюд, в которых используется продукт, фронтенд сейчас не строит, потому что этого поля нет в текущем API-контракте.

## 2. Управление блюдами

### 2.1 Создание блюда
- **Ключевое место:** `src/features/CookbookApp.tsx`, блок формы `Create Dish`; `src/cookbookApi.ts#createDish`
- **На что обратить внимание:** состав блюда задается списком строк `productId + amount`, но выбор `productId` сделан через выпадающий список продуктов, а не ручной ввод.
- **Как реализовано:** форма поддерживает название, фотографии, КБЖУ, размер порции, категорию, dietary flags и состав блюда. Для состава продукты подгружаются отдельным запросом и выбираются из `select`.
- **Почему так:** это заметно снижает шанс ошибки и делает создание блюда удобнее, чем ввод UUID вручную.

### 2.2 Автоматический расчёт КБЖУ
- **Ключевое место:** `swagger.json` (`DishResponse.autoCalculatedNutrition`), `src/features/CookbookApp.tsx`, блок деталей блюда
- **На что обратить внимание:** черновой автопересчет в форме сейчас фронтендом не выполняется.
- **Как реализовано:** фронтенд показывает основные КБЖУ блюда и отдельное поле `autoCalculatedNutrition`, если его возвращает сервер.
- **Почему так:** текущий API не предоставляет отдельного endpoint для промежуточного расчета при редактировании формы, поэтому фронтенд не может надежно пересчитывать черновые значения до сохранения.
- **Статус относительно ТЗ:** **частично реализовано**. Отображение серверного автоподсчета есть, но live-подстановка расчетных значений в форму при изменении состава не реализована.

### 2.3 Автоматическое определение категории блюда по макросам
- **Ключевое место:** отсутствует в текущем фронтенд-коде; потенциально должно быть либо в форме, либо на сервере
- **На что обратить внимание:** ни `swagger.json`, ни UI не содержат отдельной логики макросов.
- **Как реализовано:** пользователь выбирает категорию вручную в поле формы.
- **Почему так:** правило с макросами является доменной бизнес-логикой; при отсутствии явного server-side контракта безопаснее не дублировать предположения на фронтенде.
- **Статус относительно ТЗ:** **не реализовано**.

### 2.4 Управление дополнительными флагами блюда
- **Ключевое место:** `src/features/CookbookApp.tsx`, блок `Dietary flags` формы блюда; `swagger.json` (`availableFlags`)
- **На что обратить внимание:** форма пока позволяет ставить флаги вручную, без ограничения по составу.
- **Как реализовано:** текущая версия рендерит чекбоксы dietary flags и отправляет их на сервер.
- **Почему так:** в API есть поле `availableFlags` в ответе блюда, но нет механизма получения динамической доступности флагов на этапе создания/редактирования до сохранения.
- **Статус относительно ТЗ:** **частично реализовано**. Отрисовка и отправка флагов есть, но автоматическое ограничение доступности, перерасчет и автоснятие флагов в форме не выполнены.

### 2.5 Отображение блюд
- **Ключевое место:** `src/features/CookbookApp.tsx`, режим `list` для `dishes`
- **На что обратить внимание:** фильтры опираются на поддерживаемые query-параметры backend.
- **Как реализовано:** есть фильтрация по категории и dietary flags, поиск по названию и отображение списка блюд в таблице.
- **Почему так:** логика фильтрации передана backend, чтобы UI оставался простым и не зависел от объема загруженных данных.

### 2.6 Просмотр блюда
- **Ключевое место:** `src/features/CookbookApp.tsx`, режим `details` для `dishes`
- **На что обратить внимание:** в детальном просмотре показывается состав блюда с количеством.
- **Как реализовано:** по `GET /api/dishes/{id}` отображаются атрибуты блюда, КБЖУ, автоподсчитанные КБЖУ и список продуктов `productName/productId + amount`.
- **Почему так:** это соответствует требованию показывать полный состав блюда и позволяет опираться на серверный ответ как на источник истины.

### 2.7 Редактирование блюда
- **Ключевое место:** `src/features/CookbookApp.tsx`, режим `edit` для `dishes`; `src/cookbookApi.ts#updateDish`
- **На что обратить внимание:** список продуктов для выбора дополнительно подгружается при открытии create/edit формы блюда.
- **Как реализовано:** фронтенд загружает блюдо по `id`, переносит данные в форму, позволяет менять состав и сохраняет результат через `PUT /api/dishes/{id}`.
- **Почему так:** редактирование из актуальных данных снижает риск конфликта между списком и детальной записью.
- **Статус относительно ТЗ:** **частично реализовано**. Редактирование есть, но автоматический live-пересчет КБЖУ после изменения состава в форме не реализован.

### 2.8 Удаление блюда
- **Ключевое место:** `src/features/CookbookApp.tsx#removeDish`
- **На что обратить внимание:** удаление подтверждается через `window.confirm`.
- **Как реализовано:** вызывается `DELETE /api/dishes/{id}`, при успехе список перезагружается.
- **Почему так:** это минимально достаточный UX для destructive-операции без добавления дополнительной зависимости или сложного модального слоя.

## Атрибуты продукта

### АП-1. Название, фотографии, калорийность, БЖУ, состав, категория, необходимость готовки, флаги
- **Ключевое место:** `src/api.types.ts#CreateProductRequest`, `src/features/CookbookApp.tsx`, форма продукта
- **На что обратить внимание:** frontend ориентируется на фактические типы из `swagger.json`, а не на текстовую формулировку ТЗ.
- **Как реализовано:** все поля, которые есть в `CreateProductRequest` и `UpdateProductRequest`, вынесены в форму и типизированы.
- **Почему так:** контракт API является реальным техническим источником для интеграции.
- **Статус относительно ТЗ:** **частично реализовано по API-контракту**. Есть расхождение в человекочитаемых значениях категорий/типа готовности: фронтенд использует локальные словари для enum-ов, потому что backend присылает только числа.

### АП-2. Даты создания и редактирования продукта
- **Ключевое место:** `src/api.types.ts#ProductResponse`, экран деталей продукта
- **На что обратить внимание:** поля read-only и не присутствуют в форме редактирования.
- **Как реализовано:** даты отображаются только в режиме просмотра.
- **Почему так:** это соответствует ТЗ и серверному контракту: пользователь не должен изменять эти поля.

## Атрибуты блюда

### АБ-1. Название, фотографии, КБЖУ, состав, размер порции, категория, флаги
- **Ключевое место:** `src/api.types.ts#CreateDishRequest`, `src/features/CookbookApp.tsx`, форма блюда
- **На что обратить внимание:** состав блюда связан с существующими продуктами через `productId`.
- **Как реализовано:** форма поддерживает список продуктов и их количество, размер порции, категорию и прочие поля из API.
- **Почему так:** структура формы повторяет серверную модель `CreateDishRequest`.

### АБ-2. Даты создания и редактирования блюда
- **Ключевое место:** `src/api.types.ts#DishResponse`, экран деталей блюда
- **На что обратить внимание:** эти поля доступны только для чтения.
- **Как реализовано:** даты выводятся в деталке блюда и не редактируются в форме.
- **Почему так:** это соответствует требованиям и уменьшает вероятность случайного искажения системных данных.

## Правила автоматического расчета КБЖУ блюда

### ПР-1. Формулы расчета на основе продуктов
- **Ключевое место:** `swagger.json` (`autoCalculatedNutrition`), `src/features/CookbookApp.tsx`, экран деталей блюда
- **На что обратить внимание:** расчеты не продублированы на фронтенде.
- **Как реализовано:** фронтенд показывает серверное поле `autoCalculatedNutrition`, если оно пришло в `DishResponse`.
- **Почему так:** расчет КБЖУ является бизнес-логикой и должен иметь единый источник истины; в текущей архитектуре им выступает backend.
- **Статус относительно ТЗ:** **частично реализовано**. Формулы не реализованы в UI как live-черновик в форме, а только отображаются как результат работы сервера.

## Дополнительные замечания по реализации

### Обработка ошибок API
- **Ключевое место:** `src/api.http.ts`
- **На что обратить внимание:** ошибки `404`, `409` и другие читаются через `ProblemDetails`.
- **Как реализовано:** при неуспешном ответе создается `ApiError`, из которого UI достает `title/detail/status` и показывает пользователю.
- **Почему так:** это дает единый механизм обработки ошибок для всех экранов и операций.

### Типизация и привязка к Swagger
- **Ключевое место:** `src/api.types.ts`, `src/cookbookApi.ts`
- **На что обратить внимание:** не все текстовые требования ТЗ можно реализовать без поддержки со стороны API.
- **Как реализовано:** фронтенд строго следует доступным схемам и endpoint'ам из `swagger.json`.
- **Почему так:** это позволяет получить рабочий интерфейс без скрытых допущений, которые могли бы расходиться с backend.

### Список требований, которые требуют доработки backend или нового API-контракта
- **Ключевое место:** `swagger.json`
- **На что обратить внимание:** некоторые пункты ТЗ невозможно корректно завершить только силами текущего фронтенда.
- **Как реализовано:** в текущей версии они либо частично покрыты, либо зафиксированы как ограничения.
- **Почему так:** без явной поддержки API фронтенд не может надежно реализовать:
  - список блюд, блокирующих удаление продукта;
  - live-автопересчет КБЖУ блюда в форме;
  - макросы в названии блюда;
  - динамическую доступность dietary flags у блюда до сохранения;
  - файловую загрузку фотографий вместо строковых ссылок;
  - строгие ограничения из ТЗ по минимальной длине, максимуму фотографий и минимальному числу продуктов в составе, если эти правила не подтверждены API.
