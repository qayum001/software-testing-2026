# Документация по реализации ТЗ

## Введение

Этот документ описывает, как требования ТЗ были реализованы в проекте `Cookbook`.

Формат для каждого пункта:
- `Пункт ТЗ`
- `Ключевое место / на что обратить внимание`
- `Как реализовано`
- `Почему сделано именно так`

Важно:
- В текущем репозитории реализована серверная часть системы.
- Клиентское приложение (web/mobile UI) в репозитории отсутствует.
- То есть обязательная backend-часть из ТЗ реализована, но полноценный клиентский слой как отдельный интерфейс пользователя не добавлялся.

---

## 1. Нефункциональные требования

### 1.1 Архитектура системы

**Пункт ТЗ:** система должна быть реализована как web-приложение или мобильный клиент с отдельной серверной частью.

**Ключевое место / на что обратить внимание:** обязательность backend-части и отделение серверной логики от пользовательского интерфейса.

**Как реализовано:** в проекте реализован backend на ASP.NET Core Web API. Точка входа находится в `Cookbook/Program.cs`. API опубликовано через контроллеры `Cookbook/Controllers/ProductsController.cs` и `Cookbook/Controllers/DishesController.cs`. Бизнес-логика вынесена в сервисы `Cookbook/Application/Services`.

**Почему сделано именно так:** ASP.NET Core Web API хорошо подходит под серверную архитектуру с явным API-контрактом, DI, middleware и тестируемым сервисным слоем. Такое разделение упрощает дальнейшее подключение web- или mobile-клиента.

### 1.2 Хранение данных на сервере

**Пункт ТЗ:** данные о продуктах и блюдах должны храниться в базе данных или в файлах на стороне сервера.

**Ключевое место / на что обратить внимание:** допустимы как БД, так и файловое хранилище.

**Как реализовано:** выбрано файловое хранение в JSON через `Cookbook/Infrastructure/Persistence/JsonCookbookStore.cs`. Путь к файлу задается в `Cookbook/appsettings.json` и `Cookbook/appsettings.Development.json`.

**Почему сделано именно так:** файловое хранилище разрешено ТЗ и быстрее внедряется для учебного/демонстрационного решения. При этом хранилище отделено интерфейсом `ICookbookStore`, поэтому в будущем его можно заменить на реляционную БД без переписывания контроллеров и основной бизнес-логики.

### 1.3 Наличие API

**Пункт ТЗ:** серверная часть должна предоставлять API для взаимодействия.

**Ключевое место / на что обратить внимание:** API должно покрывать операции над продуктами и блюдами.

**Как реализовано:** реализованы REST endpoint'ы:
- `api/products`
- `api/dishes`

Они поддерживают создание, получение списка, получение по `id`, обновление и удаление. Контроллеры находятся в:
- `Cookbook/Controllers/ProductsController.cs`
- `Cookbook/Controllers/DishesController.cs`

**Почему сделано именно так:** REST API является самым прямым и понятным способом выполнить требование о серверном API и облегчает тестирование, интеграцию с frontend и генерацию OpenAPI/Swagger.

### 1.4 Тестируемость

**Пункт ТЗ:** код должен обеспечивать возможность написания unit-, integration-, API- и UI-тестов.

**Ключевое место / на что обратить внимание:** важна архитектура, а не только наличие тестов.

**Как реализовано:** бизнес-логика вынесена из контроллеров в сервисы `ProductService` и `DishService`, работа с данными абстрагирована интерфейсом `ICookbookStore`, а ошибки вынесены в отдельные типы исключений и middleware. Дополнительно создан отдельный тестовый проект `Cookbook.Tests`.

**Почему сделано именно так:** такая декомпозиция позволяет тестировать логику без реального HTTP и без реального файла/БД. Контроллеры остаются тонкими, а основное поведение сосредоточено в сервисах, что удобно для unit-тестов.

## 2. Управление продуктами

### 2.1 Создание продукта

**Пункт ТЗ:** пользователь должен иметь возможность создавать продукт с названием, фото, калорийностью, БЖУ, составом, категорией, необходимостью готовки и флагами.

**Ключевое место / на что обратить внимание:** обязательность основных полей и корректная модель данных.

**Как реализовано:** создан контракт `Cookbook/Contracts/Products/CreateProductRequest.cs` и модель `Cookbook/Domain/Models/Product.cs`. Создание выполняет `ProductService.CreateAsync`.

**Почему сделано именно так:** DTO запроса отделяет API-контракт от доменной модели, а сервис централизует создание и валидацию, чтобы контроллер не содержал бизнес-логику.

### 2.2 Отображение списка продуктов

**Пункт ТЗ:** список продуктов должен поддерживать фильтрацию, поиск и сортировку.

**Ключевое место / на что обратить внимание:** фильтры должны комбинироваться.

**Как реализовано:** `ProductListQuery` содержит параметры фильтрации и сортировки, а `ProductService.GetAllAsync` применяет:
- фильтр по категории
- фильтр по типу готовки
- фильтр по флагам (`Vegan`, `GlutenFree`, `SugarFree`)
- подстрочный поиск по названию
- сортировку по имени, калорийности, белкам, жирам, углеводам

**Почему сделано именно так:** все правила выборки сосредоточены в одном месте и не дублируются между контроллером и хранилищем. Это упрощает тестирование и возможную замену механизма хранения.

### 2.3 Просмотр продукта

**Пункт ТЗ:** пользователь должен иметь возможность просмотреть продукт со всеми атрибутами.

**Ключевое место / на что обратить внимание:** должен возвращаться полный набор полей.

**Как реализовано:** endpoint `GET api/products/{id}` вызывает `ProductService.GetByIdAsync`, который возвращает `ProductResponse` со всеми атрибутами продукта.

**Почему сделано именно так:** отдельный endpoint для просмотра по `id` соответствует стандартной REST-модели и упрощает интеграцию с клиентом.

### 2.4 Редактирование продукта

**Пункт ТЗ:** пользователь должен иметь возможность редактировать любые редактируемые атрибуты продукта.

**Ключевое место / на что обратить внимание:** допускается изменение всех пользовательских полей, но не системных дат напрямую.

**Как реализовано:** обновление выполняется через `PUT api/products/{id}` и `ProductService.UpdateAsync`. Обновляются имя, фото, КБЖУ, состав, категория, тип готовки, флаги. `UpdatedAt` заполняется системой.

**Почему сделано именно так:** системные поля должны оставаться под контролем сервера, иначе теряется достоверность времени изменения.

### 2.5 Удаление продукта

**Пункт ТЗ:** продукт нельзя удалять, если он используется хотя бы в одном блюде; система должна сообщить, в каких блюдах он используется.

**Ключевое место / на что обратить внимание:** это важное бизнес-правило, не просто техническая проверка.

**Как реализовано:** `ProductService.DeleteAsync` сначала ищет блюда через `ICookbookStore.GetDishesByProductIdAsync`. Если продукт используется, выбрасывается `BusinessConflictException` с данными о блюдах. Ошибка превращается в `409 Conflict` через `ApiExceptionMiddleware`.

**Почему сделано именно так:** правило относится к бизнес-логике, значит оно должно жить в сервисе, а не в контроллере и не только в UI. Это гарантирует соблюдение ограничения любым клиентом.

## 3. Управление блюдами

### 3.1 Создание блюда

**Пункт ТЗ:** пользователь должен иметь возможность создавать блюдо с названием, фото, КБЖУ, составом, размером порции, категорией и флагами.

**Ключевое место / на что обратить внимание:** состав блюда содержит ссылки на существующие продукты и количество каждого продукта в порции.

**Как реализовано:** для создания используются:
- `Cookbook/Contracts/Dishes/CreateDishRequest.cs`
- `Cookbook/Domain/Models/Dish.cs`
- `Cookbook/Domain/Models/DishProduct.cs`
- `Cookbook/Application/Services/DishService.cs`

Сервис проверяет наличие продуктов, валидность количества и наличие хотя бы одного ингредиента.

**Почему сделано именно так:** состав блюда является отдельной структурой many-to-many, поэтому он выделен в отдельную модель `DishProduct`.

### 3.2 Автоматический расчет КБЖУ

**Пункт ТЗ:** при создании и редактировании блюда система должна автоматически рассчитывать калорийность, белки, жиры и углеводы на порцию.

**Ключевое место / на что обратить внимание:** авторасчет должен основываться на КБЖУ продуктов на 100 г и количестве продукта в порции.

**Как реализовано:** в `DishService` есть метод `CalculateNutrition`, который считает:
- калории
- белки
- жиры
- углеводы

по формуле `значение_на_100г * amount / 100`. Затем метод `ResolveFinalNutrition` подставляет автозначения как базовые и позволяет вручную переопределить любое из полей, если оно было передано в запросе.

**Почему сделано именно так:** ТЗ требует именно черновые автоматически рассчитанные значения с возможностью ручной корректировки. Поэтому сервер всегда сначала считает автоматически, а потом поверх подставляет пользовательские значения только для тех полей, которые реально переданы.

### 3.3 Автоматическое определение категории блюда по макросам

**Пункт ТЗ:** макросы в названии (`!десерт`, `!суп` и т.д.) должны автоматически выставлять категорию и удаляться из названия.

**Ключевое место / на что обратить внимание:** если категория указана вручную, она должна иметь приоритет.

**Как реализовано:** `DishService.ParseNameMacro` ищет первый поддерживаемый макрос через регулярное выражение, удаляет его из названия и возвращает вычисленную категорию. Далее в сервисе используется правило:
- если `request.Category` задана, используется она
- иначе используется категория из макроса

**Почему сделано именно так:** это прямое отражение приоритета из ТЗ. Логика реализована централизованно в сервисе, чтобы правило работало одинаково и при создании, и при редактировании.

### 3.4 Управление дополнительными флагами блюда

**Пункт ТЗ:** флаги блюда доступны только если все продукты состава поддерживают соответствующий флаг.

**Ключевое место / на что обратить внимание:** при изменении состава доступность флагов должна пересчитываться, недопустимые флаги должны сниматься автоматически.

**Как реализовано:** в `DishService`:
- `GetAvailableFlags` вычисляет пересечение флагов всех продуктов в составе
- `NormalizeDishFlags` оставляет только допустимые флаги из тех, что пришли в запросе

То есть если состав изменился и ранее установленный флаг больше невалиден, он автоматически исчезает.

**Почему сделано именно так:** пересечение наборов флагов напрямую соответствует формулировке “флаг доступен только если все продукты его поддерживают”.

### 3.5 Отображение списка блюд

**Пункт ТЗ:** список блюд должен поддерживать фильтрацию по категории и флагам, а также поиск по названию.

**Ключевое место / на что обратить внимание:** фильтры должны комбинироваться.

**Как реализовано:** `DishListQuery` и `DishService.GetAllAsync` поддерживают:
- фильтр по категории
- фильтр по `Vegan`, `GlutenFree`, `SugarFree`
- подстрочный поиск по названию

**Почему сделано именно так:** логика выборки сосредоточена в одном сервисе и не зависит от конкретного транспорта данных.

### 3.6 Просмотр блюда

**Пункт ТЗ:** пользователь должен видеть все поля блюда и полный состав.

**Ключевое место / на что обратить внимание:** важно вернуть не только `productId`, но и читаемый состав.

**Как реализовано:** `DishResponse` содержит список `Products` с `ProductId`, `ProductName` и `Amount`, а также `AutoCalculatedNutrition` и `AvailableFlags`.

**Почему сделано именно так:** клиенту удобнее получать уже собранную и читаемую структуру блюда, а не делать дополнительные запросы по каждому ингредиенту.

### 3.7 Редактирование блюда

**Пункт ТЗ:** при изменении блюда КБЖУ должны автоматически пересчитываться, при этом должна оставаться возможность ручной корректировки.

**Ключевое место / на что обратить внимание:** поведение должно совпадать с созданием блюда.

**Как реализовано:** `DishService.UpdateAsync` использует ту же последовательность, что и создание:
- валидация состава
- определение категории
- авторасчет КБЖУ
- применение ручных значений поверх автозначений
- пересчет доступных флагов

**Почему сделано именно так:** дублировать или расходиться в правилах между create/update было бы ошибкой. Единая схема поведения снижает вероятность регрессий.

### 3.8 Удаление блюда

**Пункт ТЗ:** блюдо можно удалять без ограничений.

**Ключевое место / на что обратить внимание:** в отличие от продукта, ограничение отсутствует.

**Как реализовано:** `DishService.DeleteAsync` просто удаляет запись по `id` и возвращает ошибку только если блюда не существует.

**Почему сделано именно так:** это прямое следование ТЗ без добавления лишних ограничений.

## 4. Атрибуты сущностей и модель данных

### 4.1 Атрибуты продукта

**Пункт ТЗ:** у продукта должны быть название, фото, КБЖУ на 100 г, состав, категория, тип готовки, флаги, даты создания и редактирования.

**Ключевое место / на что обратить внимание:** `UpdatedAt` должен быть nullable, а перечисления должны соответствовать ТЗ.

**Как реализовано:** структура продукта задана в `Cookbook/Domain/Models/Product.cs`. Перечисления вынесены в:
- `Cookbook/Domain/Enums/ProductCategory.cs`
- `Cookbook/Domain/Enums/CookingType.cs`
- `Cookbook/Domain/Enums/DietaryFlag.cs`

`CreatedAt` выставляется при создании, `UpdatedAt` остается `null` до первого редактирования.

**Почему сделано именно так:** доменная модель должна быть достаточно строгой, чтобы ограничения ТЗ отражались в коде явно, а не только на уровне UI.

### 4.2 Атрибуты блюда

**Пункт ТЗ:** у блюда должны быть название, фото, КБЖУ на порцию, состав, размер порции, категория, флаги и системные даты.

**Ключевое место / на что обратить внимание:** состав блюда является отдельной сущностью связи.

**Как реализовано:** модель блюда описана в `Cookbook/Domain/Models/Dish.cs`, а состав в `Cookbook/Domain/Models/DishProduct.cs`.

**Почему сделано именно так:** это корректное представление отношения “многие ко многим” между блюдами и продуктами с дополнительным полем `amount`.

## 5. Валидации

### 5.1 Валидации продуктов

**Пункт ТЗ:** имя от 2 символов, КБЖУ неотрицательные, фото не более 5.

**Ключевое место / на что обратить внимание:** валидации должны выполняться на сервере.

**Как реализовано:** `ProductService.ValidateRequest` проверяет длину имени, количество фото и неотрицательность калорий/белков/жиров/углеводов.

**Почему сделано именно так:** серверная валидация защищает систему независимо от клиента, который вызывает API.

### 5.2 Валидации блюд

**Пункт ТЗ:** минимум 1 продукт в составе, `portion_size > 0`, КБЖУ неотрицательны.

**Ключевое место / на что обратить внимание:** проверяются и структура блюда, и значения вложенных ингредиентов.

**Как реализовано:** `DishService.ValidateDishCommonFields`, `ValidateComposition` и `ResolveFinalNutrition` проверяют:
- наличие хотя бы одного продукта
- положительный размер порции
- положительное количество каждого продукта
- неотрицательность переданных вручную КБЖУ

**Почему сделано именно так:** валидации разделены по зонам ответственности, чтобы код было проще читать, поддерживать и тестировать.

## 6. Обработка ошибок

### 6.1 API-ошибки и бизнес-конфликты

**Пункт ТЗ:** система должна сообщать о невозможности выполнить запрещенные операции.

**Ключевое место / на что обратить внимание:** ошибки не должны теряться или возвращаться в неструктурированном виде.

**Как реализовано:** введены исключения:
- `EntityNotFoundException`
- `ValidationException`
- `BusinessConflictException`

Они обрабатываются в `Cookbook/Middleware/ApiExceptionMiddleware.cs` и превращаются в корректные HTTP-ответы (`400`, `404`, `409`, `500`).

**Почему сделано именно так:** middleware обеспечивает единообразный формат ошибок для всех контроллеров и устраняет дублирование логики обработки.

## 7. Unit-тесты

### 7.1 Покрытие автоматического расчета калорийности блюда

**Пункт ТЗ:** требуется покрыть unit-тестами автоматический расчет калорийности блюда, используя техники тест-дизайна и инструменты unit-тестирования.

**Ключевое место / на что обратить внимание:** обязательно использовать “Эквивалентное разбиение” и “Анализ граничных значений”.

**Как реализовано:** создан проект `Cookbook.Tests` на `xUnit`. Основной тест-сьют:
- `Cookbook.Tests/DishServiceCaloriesTests.cs`

В тестах использованы:
- `Theory` + `MemberData` для параметризации
- тестовая заглушка `InMemoryCookbookStore`
- `Fact` для отдельных граничных сценариев
- XML-документация (`summary`) у тестового класса и тестов

Покрыты сценарии:
- корректные значения количества ингредиентов
- нулевое количество ингредиента
- отрицательное количество
- минимальное положительное количество
- продукт с нулевой калорийностью
- пустой состав блюда

**Почему сделано именно так:** именно сервисный слой отвечает за расчет и валидацию, поэтому unit-тестирование `DishService` дает наиболее точную и быструю проверку бизнес-логики без лишнего HTTP-шума.

## 8. Дополнительно для разработки

### 8.1 Swagger / OpenAPI

**Пункт ТЗ:** в исходном ТЗ отдельного требования на Swagger не было, но для разработки и проверки API документация полезна.

**Ключевое место / на что обратить внимание:** нужен доступ к `swagger.json`, но не ручная генерация файла.

**Как реализовано:** в `Program.cs` подключены `AddSwaggerGen()` и `UseSwagger(...)`. JSON доступен по маршруту:
- `/swagger/v1/swagger.json`

Также сохранен `OpenAPI` endpoint:
- `/openapi/v1.json`

**Почему сделано именно так:** это позволяет клиентам и инструментам получать актуальную схему API на лету без хранения статического файла, который быстро устаревает.

### 8.2 CORS для разработки

**Пункт ТЗ:** отдельного требования не было, но для локальной разработки frontend/backend это важно.

**Ключевое место / на что обратить внимание:** в dev-режиме нужен свободный доступ к API с любых origin.

**Как реализовано:** в `Program.cs` добавлена политика `AllowAll`, разрешающая любой origin, метод и заголовок.

**Почему сделано именно так:** на этапе разработки это упрощает подключение любого локального клиента. Для production такую политику нужно будет ужесточить.

## 9. Итог

Текущая реализация покрывает:
- серверную часть системы
- API для продуктов и блюд
- файловое хранение на сервере
- ключевые бизнес-правила ТЗ
- unit-тесты для автоматического расчета калорийности блюда
- Swagger/OpenAPI и CORS для удобства разработки

Текущая реализация не покрывает:
- отдельный пользовательский frontend/mobile-клиент внутри данного репозитория

Это означает, что backend-часть и основная предметная логика реализованы, а клиентский интерфейс при необходимости может быть добавлен отдельно поверх уже готового API.
